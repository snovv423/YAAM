# Схема движения данных

Простыми словами: где какие данные живут сейчас и куда это должно переехать.

```
Клиент → сайт (GitHub Pages) → backend YAAM (Node, пока не задеплоен) → SQLite/Postgres
                                        ↓                        ↓
                              Telegram-бот ресторана          ЮKassa (пока не подключена)
```

## Кто что хранит сейчас

- **GitHub** — хранит только код. Ни одного реального заказа, телефона или адреса клиента в репозитории нет и быть не должно.
- **GitHub Pages** — отдаёт статические файлы клиента (`client/`). Сам по себе ничего не хранит — это просто витрина.
- **`server/db/yaam.db`** (SQLite, локально) — реальные данные при локальном запуске: рестораны, меню, заказы, платежи. Файл в `.gitignore`, в репозиторий не попадает.
- **localStorage браузера клиента** — только техническое: выбранный город, текущая корзина, номер активного заказа, факт согласия на обработку персональных данных и его версия (оферта отдельно не сохраняется — принимается каждый раз нажатием «Оплатить»). Никогда — телефон, адрес или история заказов целиком (это должно жить на сервере, не в браузере).

## Куда это должно переехать при реальном запуске

- Backend (`server/`) — задеплоить на отдельный хостинг (обсуждали Railway; в этом запросе фигурирует Timeweb VPS — выбор за вами, архитектура не зависит от конкретного хостинга).
- База данных — при небольшом обороте текущий `node:sqlite` достаточен; при росте — миграция на PostgreSQL (схема в `server/db/schema.sql` без экзотики, миграция несложная).
- Реальные персональные данные клиентов (имя, телефон, адрес, состав заказа) — только в этой серверной базе. Не в GitHub, не в localStorage дольше одной активной сессии.

## Что получает Telegram-бот ресторана

Только данные, необходимые для выполнения конкретного заказа: имя клиента, телефон, адрес, состав заказа, комментарий, номер заказа. Бот не получает: факт согласия с офертой, версии документов, технические данные браузера.

## Что получает ЮKassa (когда будет подключена)

Только данные, необходимые для оплаты, сплита, возврата и фискализации: сумма, разбивка 7%/93%, реквизиты ресторана-субмерчанта, номер заказа как идентификатор платежа. Факт принятия оферты/согласия ЮKassa не передаётся — это внутреннее юридическое доказательство YAAM.

## Согласия (оферта / персональные данные)

Схема принята 2026-07-06: один чекбокс, а не два.

- **Оферта** — отдельного чекбокса не имеет. Акцептуется самим нажатием кнопки «Оплатить» (под кнопкой — пояснительный текст со ссылкой на `/legal/offer.html`).
- **Согласие на обработку персональных данных** — отдельный обязательный чекбокс, не отмечен заранее, без него заказ не оформить.
- Факт согласия хранится только внутри YAAM (сейчас — localStorage клиента; при появлении backend — таблица на сервере).
- **Ресторану в Telegram-боте факт согласия не передаётся** — бот получает только данные, нужные для выполнения заказа (имя, телефон, адрес, состав, комментарий).
- **ЮKassa факт согласия не передаётся**, если это не потребуется по её регламенту — она получает только данные для оплаты/сплита/возврата/фискализации.

Сейчас (frontend-only на GitHub Pages, backend не развёрнут):
- факт согласия сохраняется только в localStorage клиента (`localStorage['yaam_legal']`: acceptedPersonalData, consentVersion, privacyVersion, acceptedAt);
- никуда не отправляется, ни в один git-коммит не попадает.

Когда появится backend с базой:
- факт согласия нужно дублировать на сервере в отдельной таблице (например `legal_acceptances`: order_id, accepted_personal_data, consent_version, privacy_version, accepted_at) — localStorage слишком ненадёжен как единственное доказательство (пользователь может его очистить);
- версия принятой оферты на момент оплаты тоже стоит фиксировать в этой таблице (даже без отдельного чекбокса) — на случай споров, какая редакция действовала при заказе.
