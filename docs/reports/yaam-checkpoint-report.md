**Дата отчёта:** 7 июля 2026

## 1. Краткое описание проекта

**YAAM** — локальный агрегатор доставки еды для Чечни (Грозный, Аргун, Гудермес, Шали). Платформа соединяет клиента и ресторан: клиент оформляет и оплачивает заказ через YAAM, доставку организует сам ресторан.

**Стадия проекта:** frontend — рабочий MVP/прототип (публично доступен на GitHub Pages, работает на демо-данных); backend — написан и протестирован локально, ни разу не развёрнут на реальном сервере.

**Текущая архитектура:**
- Клиент — статический сайт (HTML/CSS/JS, без фреймворков и сборщиков), мобильный дизайн.
- Backend — Node.js + SQLite (временная БД для разработки).
- Один переключатель (`YAAM_API_BASE_URL`) определяет режим: не задан → сайт работает на демо-данных (текущее публичное состояние); задан → сайт работает через реальный API.
- Backend будет перенесён на **Timeweb VPS** с переходом на **PostgreSQL** — это следующий этап, ещё не начат.
- Приём платежей через **ЮKassa** — реализована только программная заглушка (абстракция провайдера), реальное подключение ещё не начато.

## 3. Что готово в админке YAAM

Внутренний инструмент только для команды YAAM — рестораны в него доступа не имеют (работают через Telegram-бота).

**Управление ресторанами:** список, создание, редактирование.

**Поля ресторана:** название, кухня, город(а), часы работы, доставка (₽), минимальный заказ (₽), время готовки по умолчанию (мин), телефон для заказов, фото (URL).

- **`phone_for_orders`** — в текущей схеме это поле `phone`: телефон ресторана, который клиент видит только после оформления заказа (не публикуется на карточке).
- **`image_url`** — фото ресторана/блюда хранится как URL-строка (`photo_url`). Отдельного `image_path` для файлового хранения пока нет — загрузка файлов не реализована.
- **Preview/fallback:** на клиентском сайте есть — рабочее фото или нейтральная серая заглушка «нет фото», если URL пуст/битый. **В самой админке preview отсутствует** — поле для фото сейчас просто текстовый URL без миниатюры.

**Меню и блюда:** категории, блюда (название, цена, фото, описание, состав), возможность поставить блюдо в стоп-лист одним действием.

**Стоп ресторана:** пауза с тремя пресетами (33 мин / 3 часа / 11 часов), автоматически снимается по истечении времени; ручное «Открыть сейчас» также доступно.

**Стоп-лист блюда:** переключатель «в наличии / стоп-лист» на каждое блюдо.

**Оценки/качество:** отдельный раздел — средний балл и количество оценок по каждому ресторану, лента последних оценённых заказов.

**Демо / localStorage сейчас:**
- На публичном GitHub Pages сайте (без бэкенда) рестораны, меню и голосование за новые точки — статичные демо-данные (`data.js`); в localStorage браузера хранится только согласие на обработку персональных данных и (с этой сессии) состояние корзины/активного заказа.

**Что уйдёт в PostgreSQL:** вся текущая SQLite-схема (рестораны, категории, блюда, заказы, платежи) переносится без структурных изменений — это тот же набор таблиц, просто на другой СУБД.

## 4. Что готово по рейтингам

- Оценить можно **только** заказ в статусе `delivered` — сервер отклоняет попытку оценить заказ на любом другом этапе.
- **Один `order_id` = одна оценка.** Повторная попытка оценить тот же заказ отклоняется сервером с ошибкой («вы уже оценили этот заказ») — проверено тестом: вторая оценка того же заказа возвращает `400`.
- Средний рейтинг ресторана пересчитывается атомарно при каждой новой оценке.
- **Важное уточнение:** защита сейчас работает **на уровне заказа**, а не «устройства» — то есть она даже строже, чем просто «один девайс не может оценить дважды»: заказ вообще нельзя оценить дважды никем, независимо от устройства.
- **Demo-режим (без бэкенда):** флаг «уже оценено» держится только в памяти вкладки и не сохраняется — при обновлении страницы демо-экран статуса технически можно провести через оценку заново. Это не проблема данных (в демо-режиме заказы не сохраняются нигде), но стоит иметь в виду: постоянная защита появляется только вместе с реальным бэкендом.
- **Будущая backend-защита (PostgreSQL):** переносится без изменений — те же два условия (`status = delivered`, `rating IS NULL`) в виде проверки перед `UPDATE`.

**Таблицы/поля, относящиеся к рейтингам:**
- `orders.rating` (1–5, `NULL` пока не оценено)
- `restaurants.rating` (средний балл)
- `restaurants.rating_count` (число оценок)

## 6. Что готово по backend/bot skeleton

**Что есть (написано и протестировано локально):**
- REST API: рестораны, меню, создание/статус/отмена/повтор оплаты заказа, оценка.
- Внутренняя админка (защищена basic-auth).
- Жизненный цикл заказа целиком (ожидание оплаты → оплачен → ждёт ресторан → принят → готовится → курьер → доставлен, плюс отказ/таймаут/отмена/сбой оплаты).
- Абстракция платёжного провайдера: рабочая mock-реализация + заготовка под ЮKassa с прописанным контрактом.
- Telegram-бот ресторана (принятие/отклонение заказа, выбор времени готовки, управление паузой) — код готов.

**Что ещё не развёрнуто:**
- Ни разу не запущено на реальном хостинге — только локально на машине разработки.
- Бот ни разу не подключался к реальному Telegram-токену.
- ЮKassa webhook не подключён к реальному аккаунту ЮKassa (есть только заготовка обработчика).

**Что будет на Timeweb VPS:**
- Node.js процесс backend + Telegram-бот (тот же кодовой базы, без переписывания).
- PostgreSQL вместо SQLite.
- Домен + SSL для webhook’ов (Telegram, ЮKassa).

## 12. Риски

- **Первый трафик клиентов** — главный нерешённый риск: как получить первых реальных заказчиков.
- **Качество доставки на стороне ресторана** — YAAM не управляет курьером, репутационный риск при сбоях доставки ложится на восприятие платформы.
- **Реальные данные ресторанов** — пока не собраны (см. раздел 9), без них нельзя выйти из демо-режима.
- **Юридическая проверка** — все документы черновики, без проверки юристом публичный запуск невозможен.
- **Фискализация (54-ФЗ)** — открытый вопрос, кто формирует чек; требует согласования с ЮKassa.
- **Антинакрутка рейтингов после подключения бэкенда** — текущая защита («один заказ — одна оценка, только после доставки») надёжна против повторной оценки, но не защищает от заказов, оформленных исключительно ради накрутки/понижения рейтинга; при росте платформы стоит рассмотреть дополнительные проверки (например, отметку подозрительно быстрых или дешёвых заказов).
