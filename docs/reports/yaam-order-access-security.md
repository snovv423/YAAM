# YAAM: защита доступа к заказам

Дата проверки: 13 июля 2026

## Цель

Последовательный номер `YAAM-00001` остаётся только отображаемым номером заказа и больше не является способом доступа к данным или действиям заказа.

## Реализованная модель

- Браузер перед созданием заказа генерирует два независимых 256-битных секрета через Web Crypto:
  - bearer-токен доступа к заказу;
  - ключ идемпотентности создания заказа.
- Секреты передаются только в заголовках `Authorization` и `Idempotency-Key`.
- В SQLite сохраняются только SHA-256 digest секретов.
- Ключ идемпотентности связан с SHA-256 hash нормализованного содержимого заказа. Изменённый повтор не может молча получить старый заказ.
- Повтор после потерянного HTTP-ответа возвращает тот же заказ и те же безопасные данные продолжения оплаты без создания дубля.
- Внутренний `provider_payment_id` не возвращается отдельным публичным полем.
- Получение статуса, отмена, повтор оплаты и оценка требуют правильный bearer-токен.
- Неизвестный заказ и неверная пара `номер + токен` получают одинаковый ответ `404`.
- Защищённые ответы помечаются `Cache-Control: no-store`.
- Legacy-заказы без credential-записи не получают fallback по одному номеру.
- Dev-подтверждение оплаты выключено по умолчанию и доступно только при явной комбинации `mock + local/staging + ENABLE_DEV_PAYMENT_ROUTES=true`; оно также требует токен владельца заказа.
- Backend по умолчанию слушает `127.0.0.1`; `trust proxy` допускает только `loopback`.

## Хранение в базе

Добавлены аддитивные таблицы:

- `order_access_credentials` - digest токена, digest ключа идемпотентности и hash нормализованного запроса;
- `payment_presentations` - безопасные `payment_url` и `qr_payload`, необходимые для продолжения уже созданной платёжной попытки после потерянного ответа.

Исходные bearer-токены и ключи идемпотентности сервер не хранит.

## Границы текущего этапа

Этот этап закрывает владение заказом и публичный доступ к его операциям. Он не делает текущую платёжную state machine готовой к реальным деньгам.

До подключения ЮKassa отдельно обязательны:

1. Исправление refund-логики: статус `refunded` только после подтверждённого успешного возврата провайдером.
2. Идемпотентная обработка webhook по конкретной платёжной попытке без регрессии статуса поздними событиями.
3. Crash recovery создания платежа и provider idempotency key.
4. Серверный `payment_expires_at` и правила поздней успешной оплаты.
5. XSS/CSP-аудит, поскольку bearer-токен активного заказа хранится в `localStorage`.
6. Продуктовое правило срока хранения токена после `delivered` и завершения оценки.
7. Закрытый staging с точными CORS, Nginx, firewall и env-validation до любого публичного VPS.
