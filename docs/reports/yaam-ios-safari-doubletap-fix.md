# YAAM — фикс double-tap-zoom на iOS Safari

Дата: 2026-07-09. Точечный фикс критичного мобильного UX-бага перед реальными заказами.

## 1. Проблема

На iPhone (Safari) быстрый двойной тап по кнопке «+» в карточке блюда распознавался системой как жест double-tap-zoom — страница увеличивалась, карточки становились огромными, пользоваться было неудобно. Проблема касалась потенциально всех интерактивных элементов: кнопок количества, корзины, чекаута, оплаты, табов категорий, карточек ресторана/блюда.

## 2. Причина

В `client/index.html` viewport был задан как:

```
width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no
```

Это классический приём защиты от зума, работавший до iOS 9. Начиная с **iOS 10** Safari осознанно игнорирует `user-scalable=no` (решение Apple ради accessibility — пользователь должен иметь возможность увеличить страницу, если ему это нужно). В результате тег ничего не блокировал по факту: double-tap-zoom оставался активен, а сам тег создавал ложное ощущение защиты.

Правильная защита в современном WebKit — свойство CSS `touch-action` на самих интерактивных элементах, а не глобальный флаг во viewport.

## 3. Решение

### Viewport (`client/index.html`)

```diff
- <meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
+ <meta name="viewport" content="width=device-width, initial-scale=1, viewport-fit=cover">
```

`viewport-fit=cover` добавлен для корректной работы с safe-area на iPhone с вырезом/Dynamic Island. `user-scalable=no`/`maximum-scale` убраны — они не работали и заодно блокировали доступность (пользователь с плохим зрением не мог увеличить страницу вручную).

### CSS (`client/css/style.css`)

Одно новое правило в начале файла:

```css
button,.citychip,.vote-chip,.mtab,.thumb,.card,.dish,.fulfill-chip,.rating-star{
  touch-action:manipulation;-webkit-user-select:none;user-select:none;-webkit-tap-highlight-color:transparent;
}
```

Применено к реальным классам проекта — все `<button>` (это уже покрывает «+», «−», корзину, чекаут, оплату «по СБП», «Назад», переключатель доставка/самовывоз, кнопки рейтинга) плюс кликабельные `<div>`/`<span>` без тега `button`: чипы городов, «кого ждём», табы категорий меню, миниатюры фото-галереи, карточки ресторана и блюда в списке.

`touch-action: manipulation` отключает именно double-tap-zoom и встроенную 300мс-задержку клика на этих элементах — но **не запрещает** обычный pinch-zoom страницы в остальных местах. JS не понадобился: это официально рекомендованный WebKit механизм для точно этой проблемы, без обходных путей вроде `preventDefault` на touch-событиях.

## 4. Что проверено

Через Playwright на вьюпорте 390×844 (iPhone 14/15), с полной перезагрузкой на новом порту/origin для гарантии отсутствия кэша браузера:

- `getComputedStyle(...).touchAction === 'manipulation'` подтверждён на кнопке «+», карточке ресторана, табах категорий.
- Симуляция 5 быстрых нажатий подряд на «+»: количество считается верно (1 → 3 → 6), без потерь кликов и гонок состояния.
- 4 быстрых нажатия на «−»: 6 → 2, корректно.
- `document.documentElement.scrollWidth` не изменился (390px до/после серии нажатий) — горизонтального overflow не появилось, layout не прыгает.
- Консоль браузера — 0 ошибок и предупреждений на всех шагах.

## 5. Ограничение проверки

Double-tap-zoom — нативный жест iOS Safari на движке WebKit; воспроизвести его 1-в-1 через Playwright (Chromium) нельзя — инструмент не эмулирует реальную логику зум-жестов Safari. Всё, что технически можно было проверить автоматически (применение правила, корректность счётчика при быстрых нажatиях, отсутствие overflow и ошибок), проверено и подтверждено. Финальную визуальную проверку double-tap рекомендуется сделать на живом iPhone — сам механизм (`touch-action: manipulation`) при этом является стандартным, документированным Apple/WebKit решением именно для этого сценария.
