## 1. Executive Summary

Фундамент в целом крепкий, но найдена одна критичная уязвимость в защите цены на бэкенде (обходится напрямую через API, минуя фронтенд), которую нужно закрыть до реального запуска ЮKassa. Помимо неё — один реальный визуальный баг (неверный номер заказа на экране отказа) и несколько среднеприоритетных пунктов для укрепления перед продакшеном. Архитектура состояний (корзина/активный заказ/экран) уже правильно разделена и прошла обширное тестирование в этой сессии — путаницы между «данные сохранены» и «экран должен открыться» на данный момент не найдено.

## 2. Critical Issues

### 2.1. Обход проверки цены через прямой API-запрос

В `server/services/orderService.js`, `createOrder()`:

```
const trustedItems = items.map((i) => {
  if (!i.menuItemId) return i;   // если menuItemId не передан — цена/имя НЕ проверяются
  const real = db.prepare(...).get(i.menuItemId, restaurantId);
  ...
  return { ...i, name: real.name, price: real.price };
});
```

Фронтенд (обычный UI) всегда передаёт реальный `menuItemId` — через сайт эту дыру не пробить. Но это конечная точка публичного API (`POST /api/orders`), и её можно вызвать напрямую, минуя браузер: если в запросе не указать `menuItemId`, сервер принимает `name`/`price`/`qty` от клиента без всякой проверки, и именно эта сумма попадает в оплату.

**Почему критично.** Сейчас работает с mock-провайдером (не настоящие деньги), но логика 1-в-1 перейдёт в продакшен с ЮKassa. Пример: `items:[{name:"Шашлык из баранины",price:1,qty:10,menuItemId:null}]` → ресторан получит заказ на 10 реальных шашлыков, спишется 10 ₽ вместо ~5200 ₽.

**Рекомендация (не сделано, только зафиксировано).** В `createOrder()` — если `menuItemId` отсутствует/невалиден, отклонять позицию (`throw new Error`), а не «доверять как есть». Легитимных сценариев без `menuItemId` при обращении к `/api/orders` нет ни в demo, ни в реальном режиме.

## 3. High Priority Issues

**3.1. Статичный неверный номер заказа на экране отказа.** `client/index.html`, строка 177: `<p class="rej-order-id">Заказ <span class="order-code">YAAM-00001</span></p>` — никогда не обновляется JS. На экране «отклонён/не ответил/оплата не прошла/заказ не найден» пользователь всегда видит захардкоженный `YAAM-00001`, независимо от реального номера. Тривиально чинится.

**3.2. Нет обработки возврата из фона/bfcache.** Нет ни одного обработчика `visibilitychange`/`pageshow`. Поллинг не дублируется и не течёт (единственное место присвоения таймера, всегда через `stopOrderPolling()` перед новым), но при возврате из свёрнутого состояния экран может на короткое время (до 4с, `POLL_INTERVAL_MS`) показывать устаревший статус. На мобильном Safari, который агрессивно троттлит фоновые вкладки, это может ощущаться заметнее.

**3.3. Телефон не валидируется по формату.** Ни клиент, ни бэкенд не проверяют формат `customer_phone`. Ресторан/бот получит заказ с нерабочим номером и не сможет связаться с клиентом.

## 4. Medium Priority

- CORS открыт полностью (`cors()` без опций) — нормально для demo-стадии, но перед реальным трафиком стоит ограничить origin.
- Mock-провайдер платежей теряет состояние при рестарте сервера (in-memory Map) — demo-ограничение, само исчезнет при переходе на `YookassaProvider`.
- Нет rate-limiting на `/api/orders`/`/rate` — не критично на текущем масштабе.
- `delivery_price` в админке остаётся справочным полем, нигде не участвует в реальном расчёте — уже подписано соответствующе, не баг.

## 6. State Model

Экраном управляет только `go(id)`/`popstate`-обработчик. Корзиной — `cart` + `saveCartState()`/`tryRestoreSession()`, `curRest` — это «контекст корзины», не «текущий экран» (разделено и проверено в предыдущем шаге сессии). Активным заказом — `currentOrderCode`/`saveOrderState()` + `pollOrderOnce()` (реальный API) или `restoreDemoOrder()`/`nextStatus()` (демо). Нижней панелью — `updateBar()` (видна на home/menu, только если корзина не пуста и нет активного заказа).

Правило приоритета (`active order → home → cart как нижняя панель без авто-открытия ресторана`) реализовано и протестировано. Путаницы «сохранённые данные = авто-открытие экрана» не найдено.

Слабое место: `demoStage` (`'qr'|'status'`) одновременно флаг оплаты и признак того, какой экран восстанавливать — работает корректно, но это простая строка, не отдельный enum.

## 8. Browser Lifecycle Risks

| Сценарий | Результат |
|---|---|
| Refresh с корзиной | Главная, корзина сохранена |
| Refresh с активным заказом | Восстанавливается нужный экран |
| Back/forward с активным заказом | popstate-перехват не пускает на пустой чекаут |
| Свернуть → вернуться | До 4с может быть устаревший статус (см. 3.2) |
| Битый/частично старый localStorage | Не ломает сайт — везде try/catch, явный экран «не найдено» вместо тихого зависания |

## 9. Security / Backend Trust Boundaries

| Поле | Защищено? |
|---|---|
| price / item name | Только если передан menuItemId — см. Critical 2.1 |
| item availability | Через menuItemId, та же оговорка |
| restaurant open/closed | Да, is_open проверяется |
| order status | Да, только бот двигает статус |
| rating / restaurant_id при оценке | Да, restaurant_id не принимается от клиента |
| phone | Не валидируется по формату |
| total price | Да, всегда пересчитывается сервером |

## 13. Test Results

Прогнано вживую через Playwright (демо-режим, актуальный для прода): fresh visitor — ok; корзина (add/inc/dec/refresh) — ok; двойной тап на «Оплатить» — создан ровно 1 заказ; payment-pending → refresh → восстановлен тот же экран/код; оплата → все статусы → delivered — индикатор активного заказа корректен на каждом шаге; оценка → refresh → повторная форма не показана. Ранее в этой же сессии отдельно проверены: отмена/отклонение заказа, восстановление после отмены, iPhone-viewport (390×844) без horizontal overflow, double-tap-zoom фикс.
