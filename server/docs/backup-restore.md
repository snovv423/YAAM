# Backup / restore SQLite (YAAM backend)

Закрывает Critical-пункт C2 из `yaam-timeweb-vps-readiness-audit.pdf` (отсутствие
бэкапов базы перед деплоем на VPS).

## Как это устроено

- `scripts/backup-db.js` открывает `DB_PATH` в режиме `readOnly: true` и снимает
  консистентный снимок через встроенный `node:sqlite` `backup()` — это hot-backup:
  можно снимать, пока backend работает и пишет в базу (WAL допускает
  параллельного читателя и одного писателя одновременно). Внешних зависимостей
  не требуется (не нужен `sqlite3` CLI, не нужен npm-пакет).
- Сразу после снятия бэкап переводится в `PRAGMA journal_mode = DELETE` — это
  делает файл бэкапа самодостаточным (не требует companion `-wal`/`-shm` файлов
  рядом), его можно скопировать в любое место одним файлом.
- `scripts/restore-db.js` перед перезаписью текущей базы всегда сохраняет
  safety-копию (`db/pre-restore-<timestamp>.db`) — на случай, если выбранный
  бэкап окажется не тем, что нужно.
- Ротация: по умолчанию хранятся 14 последних бэкапов (`BACKUP_RETENTION_COUNT`),
  более старые файлы удаляются автоматически при каждом новом бэкапе.

## Команды

```bash
# Снять бэкап (файл появится в server/db/backups/yaam-<timestamp>.db)
npm run backup

# Восстановить самый свежий бэкап из server/db/backups/
npm run restore

# Восстановить конкретный файл
npm run restore -- /path/to/yaam-2026-07-13T10-00-00-000Z.db
```

Переменные окружения (опционально, у каждой есть разумный default):

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `DB_PATH` | `server/db/yaam.db` | какую базу бэкапить/восстанавливать |
| `BACKUP_DIR` | `server/db/backups/` | куда класть бэкапы / откуда брать при restore без аргумента |
| `BACKUP_RETENTION_COUNT` | `14` | сколько последних бэкапов хранить |

`server/db/backups/` и safety-копии (`pre-restore-*.db`) добавлены в `.gitignore` —
это оперативные данные, не часть репозитория.

## ВАЖНО: restore требует остановленного backend

`restoreDatabase()` подменяет файл `DB_PATH` на диске напрямую. Если в это же
время в базу пишет живой процесс backend (открытое WAL-соединение), после
подмены файла его соединение окажется в рассинхроне с диском — это может
привести к ошибкам `disk I/O error` или потере записанных им данных.

**Перед restore на VPS:**
```bash
sudo systemctl stop yaam-backend
npm run restore -- /path/to/backup.db
sudo systemctl start yaam-backend
```

## Расписание на VPS (рекомендация, не часть этой задачи — не деплоится)

Ежедневный бэкап через systemd timer или cron, например:
```
0 4 * * * cd /opt/yaam/server && /usr/bin/node scripts/backup-db.js >> /var/log/yaam-backup.log 2>&1
```
Дополнительно — периодически копировать `server/db/backups/` на отдельное
хранилище (не тот же диск VPS), иначе бэкапы не спасают от отказа самого диска.
Это уже вне рамок C2 (нет второго диска/S3 на этапе аудита) и зафиксировано как
отдельный пункт в backlog.

## Проверено (round-trip)

`server/test/backupRestore.test.js`:
1. Бэкап после сидирования данных → файл содержит эти данные.
2. Порча данных (удаление записи) → `restore` из бэкапа → запись восстановлена,
   создана safety-копия испорченного состояния.
3. `restore` без явного пути берёт самый свежий файл из `BACKUP_DIR`.
4. `pruneOldBackups()` оставляет ровно `retentionCount` самых новых файлов.
5. `restore` в пустой директории без аргумента и без файлов — понятная ошибка,
   не падение с невнятным исключением.
