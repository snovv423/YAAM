# Refund state machine — архитектурные решения (Вариант A)

Итог многораундового ревью (Payment/Refund Architect, Database & Concurrency
Reviewer, Security & QA Reviewer + отдельная независимая архитектурная
критика) перед реализацией возвратов, до подключения реальной ЮKassa.

> Этот документ описывает state machine и её обоснование на **SQLite**
> (текущий production-движок). Для актуальной PostgreSQL concurrency-
> стратегии и точной карты переноса каждого места (включая refund-конвейер)
> см. [`server/docs/postgresql-concurrency-migration-matrix.md`](./postgresql-concurrency-migration-matrix.md)
> и [`server/docs/postgresql-migration-status.md`](./postgresql-migration-status.md).
> Документы не противоречат друг другу — этот описывает утверждённую бизнес-
> логику state machine (что происходит), матрица — как та же логика
> реализуется под PostgreSQL (какой SQL/технику использовать для каждого места).

## Состояния

`requested -> processing -> succeeded` (терминально) | `processing -> failed`
(терминально для этой строки, новая строка автоматически не создаётся).
`processing -> processing` — самопетля при неоднозначном исходе (throw/timeout
провайдера), с тем же `provider_idempotency_key` и экспоненциальным бэкоффом.

Изначально в обсуждении фигурировало 5-е состояние `manual_review` — от него
сознательно отказались для текущего этапа: у нас нет реального провайдера,
значит нет данных о том, какие причины отказа временные, а какие постоянные,
и строить многоступенчатую auto-retry/эскалационную политику вслепую было бы
преждевременным усложнением. Схема (partial UNIQUE-индексы, immutable-field
триггеры) при этом уже допускает больше одной строки возврата на платёж —
добавить это позже можно будет без миграции схемы.

## Почему не "просто зеркалить" payment_initial_attempts 1:1

При повторном чтении текущего кода (`orderService.js`) на этапе независимой
критики выяснилось: ни `payment_initial_attempts`, ни `payment_retry_attempts`
не имеют более 2 состояний и ни один из них не имеет автономного фонового
sweep, который бы сам перезапускал неоднозначные попытки по расписанию — весь
retry там инициируется кликом пользователя (`retryPaymentFlow` в
`client/js/app.js`). Единственные `setInterval` в `server.js` — это
`sweepTimeouts` (SLA заказа, не платежа) и `sweepPauseExpiry`. Введение
многоступенчатого auto-spawning retry для возвратов было бы новым для кодовой
базы паттерном без крайней необходимости — поэтому выбран более простой,
однострочный (без порождения новых строк) вариант.

## Обязательная атомарность: бизнес-переход + резервация возврата

Нельзя делать `orders.status = X` → COMMIT → отдельным шагом (fire-and-forget)
резервировать возврат: между этими двумя шагами есть crash-window, где заказ
уже, например, `cancelled`, а обязательство "деньги нужно вернуть" нигде не
записано. Поэтому `reserveRefundRow()` — bare-функция без собственной
транзакции, вызываемая только изнутри уже открытой
`db.immediateTransaction()` в `cancelByCustomer`/`restaurantDecline`/
`sweepTimeouts`, вместе с переходом статуса заказа. Сетевой вызов провайдера
выполняется только после COMMIT (`scheduleRefundProcessing`, fire-and-forget)
— отмена заказа не блокируется на сетевом round-trip к провайдеру; клиент
узнаёт результат через `order.refund_status` при следующем poll.

## Устойчивость к падению процесса во время сетевого вызова

`next_attempt_at` выставляется в момент claim (`requested`/`processing` ->
`processing`), **до** `await` сетевого вызова, а не постфактум при пойманной
ошибке. Если процесс упадёт прямо во время ожидания ответа провайдера,
`sweepStuckRefunds()` после рестарта всё равно найдёт строку по истёкшему
дедлайну и безопасно повторит попытку тем же `provider_idempotency_key`.
Это отличается от более раннего черновика плана, где дедлайн ставился только
по факту пойманного throw/timeout — там оставалось окно: падение процесса
ровно во время сетевого вызова никогда не выставляло бы дедлайн, и такая
строка зависла бы бессрочно до ручного вмешательства.

## Публичный контракт

`GET /api/orders/:code` отдаёт `refund_status` в узком клиентском словаре
(`none | processing | done | failed`), не 1:1 зеркало внутренних состояний —
чтобы клиент не завязывался на появление будущих промежуточных внутренних
состояний.

## Явно вне рамок текущего этапа

- Автоматическое порождение новой строки возврата после явного `failed`.
- Реальная production-валидация ЮKassa и `YookassaProvider.verifyWebhook()`
  (см. примечание о фактическом состоянии провайдера ниже).
- Частичные возвраты (полный возврат — единственный сценарий MVP, обеспечено
  триггером `trg_refunds_amount_matches_payment`).
- Админская ручная отмена/возврат — отдельная будущая задача.

## Примечание: фактическое состояние YookassaProvider (актуализировано)

На момент первой редакции этого документа `YookassaProvider.refund()` был
заглушкой. С тех пор (отдельные, более поздние задачи, не эта) реализованы:
`createPayment()`, `getStatus()`, `refund()`, `getRefund()` — все с реальными
HTTP-вызовами к API ЮKassa, Idempotence-Key, обработкой таймаутов/ошибок и
тестами. Единственный метод, который на сегодня по-прежнему бросает
`not implemented`, — `verifyWebhook()` (см. `server/services/
paymentProviders/yookassaProvider.js`; при расхождении с этим файлом
источником истины считать код).
