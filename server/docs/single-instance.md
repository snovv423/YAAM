# Единственный экземпляр backend (single instance)

Закрывает Critical-пункт C3 из `yaam-timeweb-vps-readiness-audit.pdf`.

## Почему это обязательное ограничение, а не рекомендация

YAAM backend спроектирован как один процесс, и это не случайность, которую
можно "починить" масштабированием позже без переписывания логики:

1. **`orderService.createOrder()`** — уникальность `public_code` (формат
   `YAAM-00001`) и вставка заказа+платежа гарантированно атомарны только
   потому, что `node:sqlite` синхронен и вся функция выполняется одним блоком
   без интерливинга с другим вызовом `createOrder()` **в том же процессе**.
   Два процесса, пишущие в один файл SQLite одновременно, этой гарантии не дают.
2. **`sweepTimeouts()`/`sweepPauseExpiry()`** (`server.js`) — периодические
   `setInterval`, которые переводят просроченные заказы в `timed_out` и снимают
   истёкшие паузы ресторанов. Это in-process таймеры: два живых процесса —
   это два независимых, ничего не знающих друг о друге автосвипа, дублирующих
   работу (и потенциально гоняющихся за одними и теми же строками).
3. **SQLite сама по себе** не рассчитана на нескольких активных писателей с
   высокой конкурентностью так, как это делает клиент-серверная СУБД — при
   текущем масштабе (один VPS, один процесс) это осознанное упрощение
   (см. `CLAUDE.md`: "миграция на PostgreSQL — в планах"), но оно работает
   только пока экземпляр действительно один.

## Как это гарантируется на рантайме

`server/singleInstanceLock.js` — при старте `server.js` вызывает `acquireLock()`
**до** `app.listen()`:

- Если PID-lock файл (`server/db/yaam-backend.pid` по умолчанию,
  переопределяется `SINGLE_INSTANCE_LOCK_PATH`) существует и записанный там PID
  всё ещё жив (`process.kill(pid, 0)`) — процесс сразу завершается с
  `process.exit(1)` и понятным сообщением в лог, **не** доходя до `app.listen()`.
- Если lock-файла нет, либо PID в нём уже мёртв (процесс упал, не успев
  освободить lock) — новый процесс забирает lock (перезаписывает своим PID) и
  стартует нормально.
- При штатной остановке (`SIGTERM` — то, что шлёт `systemctl stop`, и `SIGINT`
  — Ctrl+C) lock освобождается перед выходом.

Это runtime-проверка, а не просто договорённость в документации: `systemctl start`
дважды подряд, забытый `npm start` в соседнем терминале, deploy-скрипт,
случайно запущенный повторно — во всех этих случаях второй процесс откажется
стартовать сам, без участия человека.

## systemd

`server/deploy/yaam-backend.service` (заготовка, на VPS не применяется этой
задачей) — намеренно `Type=simple`, без кластеризации/шаблонного юнита с
несколькими воркерами. Один `ExecStart`, `Restart=on-failure` — если процесс
упадёт, systemd поднимет новый (и он сам заберёт стал lock, так как предыдущий
уже мёртв).

## Проверено вручную (не только юнит-тестами)

Живой прогон (Bash, вне рамок Timeweb — на локальной машине):
1. `node server.js` — стартует, слушает порт, создаёт PID-lock.
2. Второй `node server.js` с тем же `SINGLE_INSTANCE_LOCK_PATH`/`DB_PATH` —
   немедленно завершается с кодом 1 и сообщением
   `Обнаружен уже запущенный процесс backend (pid=..., lock-файл: ...)`.
3. `kill -TERM` первому процессу — лог показывает штатное завершение, lock-файл
   удаляется в пределах секунды.
4. Третий `node server.js` после этого — стартует нормально (lock свободен).

Автоматизировано в `server/test/singleInstanceLock.test.js` (10 тестов):
acquire без конфликта, acquire с живым конфликтующим PID (бросает исключение),
acquire со stale (мёртвым) PID (подхватывает), создание вложенной директории
под lock-файл, release только для владельца лока, release не трогает чужой
лок, release на несуществующем файле не падает, `isProcessAlive()` для живого/
несуществующего/некорректного PID.
