# PostgreSQL admin port — server/routes/postgresql/admin.js (Production Switch Stage 4)

Подробности, для которых не хватило места в `postgresql-migration-status.md`.
Компактная сводка там же, ссылка сюда.

## Архитектурная граница

Модуль **не импортируется** ни из `server.js`, ни из `bot/index.js`, ни из
`routes/admin.js` (SQLite). SQLite-admin не изменён ни на строку. Basic Auth
здесь НЕ реализован — как и в SQLite-оригинале, он живёт исключительно в
`server.js`, в точке монтирования (`app.use('/admin', basicAuth({...}),
adminRoutes)`). Роутер, как и оригинал, авторизацию-агностичен по
конструкции. Тесты Stage 4 воспроизводят ТУ ЖЕ схему монтирования в
собственном тестовом Express-приложении, не трогая `server.js`.

`server/admin/layout.js` переиспользуется НАПРЯМУЮ (не дублируется) — чистая,
безсайд-эффектная HTML-шаблонная функция без зависимости от БД, тот же
принцип, каким Stage 3 переиспользовал класс `TelegramBot`.

## НАЙДЕННЫЙ, СЕРЬЁЗНЫЙ, УНАСЛЕДОВАННЫЙ ОТ SQLite БАГ: редактирование ресторана недостижимо

Самая важная находка Stage 4. `routes/admin.js` регистрирует:

```js
router.post('/restaurants/:id/', (req, res) => res.redirect(307, `/admin/restaurants/${req.params.id}`));
router.post('/restaurants/:id', (req, res) => { /* реальный UPDATE */ });
```

Под Express `strict: false` (значение по умолчанию, не переопределено ни в
оригинале, ни в порте) оба паттерна — `/restaurants/:id/` и
`/restaurants/:id` — компилируются `path-to-regexp` в **идентичный** регэксп
(с опциональным trailing slash). Живой мини-репродукт на чистом Express
(без единой строки кода этого проекта) подтвердил эмпирически: запрос БЕЗ
trailing slash (ровно то, что шлёт форма редактирования, `action="/admin/
restaurants/${r.id}"`) матчит **первый зарегистрированный** маршрут —
307-редирект-заглушку, а не реальный UPDATE. 307 сохраняет HTTP-метод при
редиректе — редирект ведёт на ТОТ ЖЕ САМЫЙ URL, то есть реальный браузер
получил бы **бесконечный редирект-цикл** при попытке сохранить изменения
ресторана.

Это баг **SQLite-оригинала**, не привнесённый Stage 4 — подтверждено тем,
что регистрация маршрутов в PostgreSQL-порте является буквальной,
построчной копией SQLite-файла (тот же порядок, тот же паттерн). Задание
прямо требует "сохранить текущее поведение максимально точно" и не
перечисляет исправление багов маршрутизации среди целей Stage 4 —
переставлять порядок регистрации было бы незапрошенным изменением продукта.
**Порт заведомо и намеренно воспроизводит это поведение один в один**, а не
исправляет.

Тест `C3` живо доказывает: PostgreSQL-порт ведёт себя идентично (307, не
302, данные не обновляются) — то есть перенос выполнен корректно. Тест `C3b`
отдельно доказывает, что сам SQL реального UPDATE-обработчика написан
правильно (прямой вызов, минуя маршрутизацию) — баг чисто в порядке
регистрации express-роутов, не в бизнес-логике/SQL.

**Рекомендация (НЕ выполнена в рамках Stage 4— вне мандата "не менять
продуктовую семантику"):** поменять местами регистрацию двух маршрутов
(зарегистрировать `/restaurants/:id` ПЕРЕД `/restaurants/:id/`) устранило бы
баг в ОБЕИХ версиях (SQLite и PostgreSQL) без единого другого изменения —
это тривиальный однострочный фикс, но требует отдельного явного решения
(затрагивает продуктовое поведение живого SQLite-admin, вне scope этой
изолированной задачи).

## Дialect differences (SQL и DTO-совместимость)

1. **COUNT/SUM → bigint-as-string.** `pg` по умолчанию возвращает `bigint`
   строкой. Все `COUNT(*)`/`COUNT(o.id)`/`SUM(...)` в дашборде явно приведены
   `::int`. Проверено прямым `typeof`-тестом (`B2`, `G3`), не только
   визуальным совпадением в HTML (строка `"5"` и число `5` рендерятся
   одинаково — визуальное совпадение НЕ доказывает отсутствие проблемы).
2. **MAX не требует каста** — `MAX(int4)` в PostgreSQL остаётся `int4`, не
   продвигается до `bigint` (в отличие от `SUM`/`COUNT`) — подтверждено, не
   давало сбоев без явного каста.
3. **TIMESTAMPTZ → нативный JS Date, не строка.** SQLite хранит
   `created_at`/`paused_until` как TEXT `"YYYY-MM-DD HH:MM:SS"` (UTC),
   оригинал интерполирует эту строку напрямую. `pg` возвращает такие
   колонки как настоящие `Date`-объекты — `String(date)` дал бы длинный
   локализованный формат, ломая визуальный контракт. Добавлен
   `formatDateTime(date)`, воспроизводящий тот же `"YYYY-MM-DD HH:MM:SS"`
   (UTC) формат из `Date`. `statusBadge()` использует тот же `Date` напрямую
   (`.getHours()`/`.getMinutes()`, локальное время сервера — то же
   намерение, что и у оригинала, который парсит строку с `.replace(' ','T')
   +'Z'` перед тем же вызовом) — без промежуточного строкового парсинга,
   которое здесь просто не нужно.
4. **`date('now')` → `AT TIME ZONE 'UTC'`.** SQLite `date(created_at) =
   date('now')` всегда сравнивает UTC-даты (SQLite не имеет понятия
   часового пояса). PostgreSQL-эквивалент явно приводит обе стороны к UTC
   (`(created_at AT TIME ZONE 'UTC')::date = (NOW() AT TIME ZONE
   'UTC')::date`), не полагаясь на session timezone пула соединений.
5. **`lastInsertRowid` → `RETURNING id`.** `POST /restaurants` использует
   `INSERT ... RETURNING id` вместо `.lastInsertRowid`.
6. **`GROUP BY r.id` без перечисления остальных `r.*` колонок** — валиден в
   PostgreSQL по функциональной зависимости через PRIMARY KEY (тот же
   паттерн уже живо проверен в `routes/postgresql/api.js`, Stage 1) — не
   требует `GROUP BY r.id, r.name, r.rating, r.rating_count`.
7. **FK-нарушения** (например, категория/блюдо для несуществующего
   ресторана) перехватываются ОБЩИМ `catch`, не специальным
   404-обработчиком — SQLite-оригинал ТОЖЕ имеет `PRAGMA foreign_keys = ON`
   (проверено в `db/index.js`) и точно так же не обрабатывает этот случай
   отдельно (та же категория ошибки, что и в оригинале, просто с чистым
   сообщением вместо трассировки — задание, раздел "Безопасность", п.5).
8. **REAL (rating)** — `pg` возвращает `float4`/`real` как обычный JS
   `number`, без строковой защиты (в отличие от `numeric`/`bigint`) —
   `.toFixed(1)` работает без изменений.
9. **is_open/is_available остаются INTEGER 0/1 в обеих схемах** — PostgreSQL-
   схема сознательно НЕ использует нативный `BOOLEAN` для этих полей именно
   ради полной совместимости с SQLite-конвенцией (см. комментарий в
   `db/postgresql/schema.sql`) — предполагавшееся заданием "BOOLEAN
   возвращается как true/false вместо 0/1" в этом конкретном случае НЕ
   применимо, диалектного расхождения нет.

## Concurrency: сужение read-modify-write гонки для sort_order

SQLite-оригинал выполняет `SELECT MAX(sort_order)` и `INSERT` ДВУМЯ
отдельными синхронными вызовами. Под однопоточным SQLite это не создаёт
гонки — Node физически не может переключиться на другой запрос между двумя
синхронными операциями одного request-обработчика. Под PostgreSQL (async)
буквальный перенос ("await SELECT MAX", затем "await INSERT") создал бы
РЕАЛЬНОЕ окно: два конкурентных добавления категории/блюда для ОДНОГО
ресторана могли бы прочитать одинаковый MAX и получить одинаковый
`sort_order`.

**Решение:** `MAX+1` вычисляется ВНУТРИ самого `INSERT` одним SQL-
выражением (`VALUES ($1, $2, (SELECT COALESCE(MAX(sort_order),0)+1 FROM ...
WHERE restaurant_id = $1))`) — один round-trip вместо двух. Это СУЖАЕТ окно
гонки практически до нуля для реалистичного admin-использования (два штатных
клика), но теоретически НЕ устраняет его полностью без `SERIALIZABLE` или
блокировки, которые задание прямо просит не вводить автоматически. Цена
проигрыша этой узкой гонки — исключительно косметическая: два элемента с
одинаковым `sort_order`, порядок между ними произволен, отображение остаётся
стабильным при повторной сортировке (естественный tie-break по `id`). Не
нарушение данных, не ошибка пользователю, не потеря строки. Живо доказано
`D6`/`E8` (Promise.all на создание) — обе строки создаются, ничего не
теряется, конкретное значение `sort_order` не детерминировано и не
проверяется строго.

`toggle-available` использует УЖЕ доказанный безопасным паттерн (Stage 3,
bot-стоп-лист) — атомарная арифметика `is_available = 1 - is_available`
целиком внутри `UPDATE`, без предварительного чтения в JS — построчная
блокировка PostgreSQL полностью сериализует конкурентные вызовы, без
`lost update`. Живо доказано `E5` и отдельным 20-итерационным стресс-
прогоном.

## Security

- Basic Auth сохранён полностью (тестовый harness воспроизводит ТУ ЖЕ схему
  монтирования, что и `server.js`) — `A1`-`A4`.
- SQL-инъекция исключена конструктивно — все запросы параметризованы
  (`$1,$2...`), пользовательский ввод никогда не конкатенируется в SQL-текст.
- Ошибки PostgreSQL никогда не отдаются клиенту сырым текстом — общий
  `try/catch` на каждом маршруте, фиксированное сообщение "Внутренняя ошибка
  сервера" (`C5`, `C6`, `C7`, `D3`, `D4`).
- Секреты (provider keys, order access token hashes, idempotency keys) НЕ
  читаются и НЕ выводятся ни одним admin-маршрутом — ни в SQLite-оригинале,
  ни в порте (подтверждено построчным аудитом — эти данные вообще не
  затрагиваются admin-роутами).
- "Владение" ресторан/блюдо — концептуально НЕ применимо к этому admin-
  панели: это ОДИН внутренний инструмент YAAM с ОДНОЙ общей учётной записью
  для всех ресторанов (не портал самообслуживания, см. CLAUDE.md), поэтому
  "чужой ресторан не может управлять чужим блюдом" — не применимая для этого
  уровня модель угроз (тот же принцип уже подтверждён для SQLite-оригинала).
- Числовые поля (`price`/`delivery_price`/`min_order`/`default_cook_minutes`)
  нормализуются той же схемой, что и оригинал (`Number(x) || 0` — тот же
  fallback-паттерн, дословно) — не изменено.
- CSRF-поведение не изменено относительно оригинала (ни оригинал, ни порт не
  реализуют CSRF-токены — не входит в scope переноса).

## Документированный найденный маршрутный баг — итог

Помимо баг с `/restaurants/:id/` (описан выше), отдельных новых security-
находок сверх уже задокументированных в Stage 3 (отсутствие ownership-
проверки, не-`UNIQUE` `telegram_chat_id`) не обнаружено — admin-панель, в
отличие от бота, не имеет per-restaurant identity-модели вообще (единая
учётная запись на всю панель), поэтому класс проблем "чужой ресторан
управляет чужими данными" к ней структурно неприменим.
